# CardVault Frontend Implementation Plan

## Goal

Build a production-ready frontend ecosystem for CardVault using:

- Mobile app: React Native / Expo for iOS and Android
- Web app: Next.js
- Shared code: TypeScript packages for API client, types, validation, utilities, and scan parsing
- Backend: Existing CardVault Go API

The mobile app will focus on collection management and real-time card scanning.  
The web app will focus on collection management, analytics, watchlists, dashboards, and sharing.

## Technical Direction

Use a monorepo with separate mobile and web apps.

Recommended stack:

- Package manager: `pnpm`
- Monorepo orchestration: `Turborepo`
- Mobile: `Expo + React Native`
- Web: `Next.js App Router`
- Language: `TypeScript`
- API data fetching: `TanStack Query`
- Forms: `React Hook Form`
- Validation: `Zod`
- Mobile navigation: `Expo Router`
- Web styling: Tailwind CSS
- Mobile styling: NativeWind or React Native StyleSheet
- Shared API client: custom typed fetch client
- OCR/scanning: React Native VisionCamera + ML Kit/VisionCamera OCR plugin

Turborepo is suitable here because it is designed for JavaScript/TypeScript monorepos and shared package workflows. Next.js App Router is the current recommended Next.js routing model, and VisionCamera supports frame processors for real-time camera workloads.

Mobile Scan Flow

Implement real-time scanning, not photo capture.

Flow:

Open scan screen
→ Request camera permission
→ Start camera preview
→ Process frames at controlled interval
→ Run OCR locally
→ Parse OCR text into hints
→ Wait for stable hints across multiple reads
→ Call backend /scan/resolve
→ Show exact result or candidate list
→ User confirms
→ Add card to collection

Do not run OCR on every frame. Use throttling.

Recommended behavior:

Process one frame every 300–500ms
Avoid parallel OCR jobs
Use a central overlay region
Only resolve when hints are stable across 2–3 readings
Stop scanning while backend request is in progress
Resume scan if result is rejected

VisionCamera is appropriate because it supports real-time frame processors.

Mobile Scan States

Implement explicit states:

type ScanState =
  | 'idle'
  | 'requesting_permission'
  | 'camera_ready'
  | 'scanning'
  | 'processing_frame'
  | 'resolving'
  | 'exact_match'
  | 'multiple_candidates'
  | 'not_found'
  | 'error';

First Implementation Target

Start with this order:

Monorepo setup
Shared API client
Auth on mobile and web
Collection list on mobile and web
Mobile scan screen skeleton
OCR integration
Backend scan resolve integration

Do not start with advanced OCR. First build the full vertical slice:

camera preview
→ fake OCR text
→ parse hints
→ call backend
→ show candidates
→ add to collection

After the vertical slice works, replace fake OCR with real frame OCR.

Important Product Decisions
Real-time scan is mobile-only.
Web is for management, analytics, and sharing.
OCR is not the source of truth.
Backend scan resolution is the source of truth.
Shared code should contain logic, not platform UI.
The scan flow must always have manual fallback.
Avoid overengineering visual recognition in the first version.
